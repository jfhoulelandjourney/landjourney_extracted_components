# Extraction plan

The persistent plan-of-record for this repo. Updated at the end of every wave.
A separate progress file (`EXTRACTION_PROGRESS.md`) is auto-generated and tracks
every component vs. whether it has a story.

## Goal

Mirror the **real** Angular components from
`/Users/jfhoule/Documents/Code/2026/lanjourney_local_environement_2026/user-interfaces/projects/common/src/lib/`
into this repo as a Storybook playground, so the design quality of every UI
component can be browsed and improved in isolation — without running the
production app and without modifying the source repo.

## Hard constraints

- The source repo is **read-only**. Components are physically **copied** in
  (no submodule, no npm-link, no path-alias). Re-syncing later means re-copying.
- Stack must mirror source: **Angular 21**, standalone components, OnPush,
  signals, NgRx Signals, SCSS, Angular Material 21.
- Components must be **the real source files**, untouched. Only data
  dependencies (services, stores, route, dialog refs) are stubbed at the story
  layer.
- Source repo's existing `.stories.ts` files (the ~10 the team had written)
  are restored verbatim — they're the canonical pattern.

## Working patterns

### Where to put a story

Mirror the source path. `<lib-root>/<area>/<component>/<file>.component.ts`
gets a sibling `<file>.component.stories.ts`. Storybook title uses the same
hierarchy: `'Web Components/Form/Money Input Field'`,
`'Design System/Molecules/Avatar'`, etc.

### Story style

Default + 3–6 named variants per component. Match the depth of the existing
button/avatar source stories for hero components; keep primitives leaner.
Always include realistic mock data — borrower names, loan amounts, dates —
not `'Lorem ipsum'`. The point of this repo is to see the components rendered
the way they really look.

### Service / store stubs

Cast through `unknown as ServiceType` (NOT `Partial<ServiceType>` — Partial
breaks on private members and inferred types):

```ts
const documentServiceStub = {
  getFileMetadata: (_id: string, _digest: string) => of({}),
  getFileTemplateMetadata: (_id: string) => of({}),
} as unknown as DocumentService;

decorators: [
  applicationConfig({
    providers: [
      { provide: DocumentService, useValue: documentServiceStub },
    ],
  }),
],
```

`provideAnimations`, `provideRouter([])`, and `provideLottieOptions` are
already wired in `.storybook/preview.ts` — don't re-provide them per story.

### Dialog stories

Render the dialog component **inline** inside a styled wrapper so the body and
actions are visible without invoking `MatDialog.open()`. Provide
`MAT_DIALOG_DATA` and a stubbed `MatDialogRef` via `applicationConfig`. Use the
existing `general-message-dialog` story as the template.

### Mock data shape

When a model is large (e.g. `DetailedLoanCompoundSchema` has ~30 required
fields), build a typed factory:

```ts
const overdueLoan = (
  partial: Partial<LoanOverviewSchema> & { id: string; name: string }
): LoanOverviewSchema =>
  ({ /* sane defaults */, ...partial } as unknown as LoanOverviewSchema);
```

Don't laboriously fill every field — `as unknown as Type` is fine for stories.

### When a component is too heavy

If a component pulls 5+ services or domain-coupled types that explode the mock
surface, **defer to a later wave** and note it in the wave's deferred list.
Do not block on a single component.

## Resuming after a break (any agent — Claude, other)

The exact protocol any agent must follow at the start of every new session is
in **`CLAUDE.md` / `AGENTS.md`** at the repo root. Both files are mirrors of
the same content. Short version:

1. **Sync the source repo.** Run `npm run sync-source`. The source codebase
   evolves — components get added, models grow new fields. The sync script
   walks `<source>/projects/common/src/lib/` and copies any file that's NOT
   already in this repo. It never overwrites local files and never deletes.
   Output reports new components needing stories + orphans (files in this
   repo no longer in source — flag for human review).
2. **Refresh the checklist.** `npm run list-components` rewrites
   `EXTRACTION_PROGRESS.md` with the current state.
3. **Read this file (`PLAN.md`).** Look at "Status by wave" to know where
   the previous session stopped, what's deferred, and why.
4. **Look at recent commits.** `git log --oneline -20` shows the last
   handful of waves so you know what landed.
5. **Pick up the next wave.** Use the patterns under "Working patterns"
   verbatim. Don't reinvent service-stub or dialog-story shapes.
6. **End-of-session: update this file.** Move the wave you finished from
   ⏳ to ✅, list new stories added, update deferred items. The next session
   reads this to know where to start; if you don't update it, work gets
   re-done.

The `CLAUDE.md` / `AGENTS.md` files contain the full agent runbook (common
pitfalls, "things that are not optional," etc.). Keep them in sync — they're
duplicated on purpose so different agent runtimes (Claude Code, Cursor,
others that read AGENTS.md) get the same guidance.
